# 基于 RAG + Agent 的智能知识库问答系统 —— 设计文档

## 1. 项目概述

一个具备自主决策能力的智能知识库问答 Agent。用户上传私有文档后，系统自动构建向量知识库；用户以自然语言提问时，系统通过双语语义检索 + LLM 自主判断的混合策略生成回答——相关时引用原文，不相关时标注"由大模型直接生成"后直接回答。

**核心价值**：解决传统意图分类做路由时的高误判问题，让 LLM 自行判断检索结果的相关性，用一个 Prompt 统一处理"查库"和"闲聊"两种场景。

## 3. 核心技术决策

### 决策一：取消意图分类，改为检索优先 + LLM 自主判断

**背景**：业界常见的 RAG 系统通常先做意图分类（knowledge/chat），再走不同分支。早期版本也采用了此方案。

**问题**：意图分类误判率高。例如"你是什么大语言模型"被误判为 knowledge，进入 RAG 流程后知识库无相关内容，返回"未找到相关信息"——用户期待的是一个直接回答。

**方案**：废弃意图分类器，所有请求统一走"检索 → 注入 Prompt → LLM 自主判断"的单一路径。Prompt 中明确指示：检索结果相关则引用，不相关则用自己的知识回答并附加来源声明。

**效果**：消除了意图分类误判导致的用户体验断裂，同时简化了系统架构（移除了 2 个处理分支和 1 个分类器模块）。

### 决策二：双语并行检索策略

**背景**：用户用中文提问（"阅读理解 Section C 主要讲了什么"），而知识库中存储的是英文文档（六级真题 PDF）。单一语言的语义检索存在跨语言匹配衰减。

**方案**：
1. 用 DeepSeek 将中文查询翻译为英文检索关键词
2. 分别以中、英文各检索一次
3. 按 page_content 去重合并，中文结果优先排列
4. 合并后的结果注入 Prompt

**效果**：相比纯中文检索，双语策略在英文文档场景下命中率显著提升。例如"CET-6 阅读 Section C"的 Passage One 和 Passage Two 内容在双语检索后均能被正确找到并总结。

### 决策三：放弃固定相似度阈值，让 LLM 自主判断

**背景**：尝试过用相似度分数（如 > 0.3 视为相关）做硬路由。

**问题**：ChromaDB 使用 L2 距离，不同查询的分数分布极度压缩（0.03 ~ 0.06），无法划定一个可靠阈值。同类查询在不同模型下分数范围差异巨大。

**方案**：不做分数过滤，将全部检索结果交给 LLM 自行判断。Prompt 中明确指示判断标准和行为准则。

**效果**：LLM 比固定阈值更智能——它能理解"宇宙是什么"与通信技术文档不相关，也能识别"Section C 的主旨"与 Page 6-7 的 Passage One/Two 文本有关联。

### 决策四：Prompt 设计的迭代演进

| 版本 | Prompt 核心指令 | 问题 |
|------|----------------|------|
| v1 | "只有完全不相关时才说找不到" | LLM 倾向于保守，频繁返回"未找到" |
| v2 | "不相关则自己回答，加免责声明" | LLM 会用"抱歉无法回答"敷衍 |
| v3 | "必须给出实质性回答，绝对不能说'无法回答'" | 同时明确"引用了参考片段就不加声明" |

**心得**：Prompt Engineering 不是一次性工作，需要根据 LLM 的实际输出反向调试。措辞的细微差异（"可以自己回答" vs "必须给出实质性回答"）会导致完全不同的行为。


## 4. 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| LLM | DeepSeek API | 生成与查询翻译 |
| 嵌入模型 | paraphrase-multilingual-MiniLM-L12-v2 | 支持 50+ 语言，本地 CPU 推理 |
| 向量数据库 | ChromaDB | 轻量级持久化存储 |
| 框架 | LangChain（Document / Embeddings / Chroma / TextSplitter） | 标准化组件接口 |
| 前端 | Streamlit | 对话式 Web 界面 |
| 文档解析 | PyPDF + pdfplumber + docx2txt | 多格式支持 |
