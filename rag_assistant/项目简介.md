## 先说一个最根本的问题：这个项目到底在干什么？

想象一个场景：你有一本 500 页的《人工智能入门》教材。

- **传统方式**：你想知道"什么是神经网络"→ 你翻目录 → 找到第 200 页 → 自己读 → 自己总结
- **你做的系统**：你把教材上传进去 → 系统自动"消化" → 你直接问"什么是神经网络" → 它告诉你在第 200 页有相关内容，并帮你总结好

这就是 **RAG（检索增强生成）** 的核心价值——让 AI 基于你的私有文件回答问题，而不是瞎编。

### 1. LLM（大语言模型）是什么？

**一句话**：LLM 就是一个"超级读了很多书的聊天机器人"。

它读了互联网上几乎所有的文章、书籍、代码，所以你跟它聊什么它都能接上话。但它有两个致命问题：
- 它的知识截止到训练时间（比如 2024 年），之后的不知道
- 它不知道你电脑里的私有文件（比如你公司内部的文档）

你项目里用的 DeepSeek 就是一个 LLM，相当于"大脑"。

### 2. 什么是 Prompt？

**一句话**：Prompt 就是你跟 LLM 说的那段话。

比如你问"什么是机器学习"，"什么是机器学习"就是 Prompt。

但你项目里的 Prompt 要复杂得多——你不会直接把问题发给 LLM，而是先查知识库，把查到的答案塞进 Prompt 里，然后再发给 LLM。这就是 **Prompt Engineering（提示词工程）**。

你在 `retriever.py` 里写的 `build_prompt` 函数，就在做这件事——组织一段包含检索结果的 Prompt。

### 3. 什么是 RAG（检索增强生成）？

**RAG = 检索 + 生成**

- **检索（Retrieval）**：用户问"什么是神经网络"→ 去知识库里搜，找到最相关的几个段落
- **增强（Augmented）**：把搜到的段落塞进 Prompt
- **生成（Generation）**：把 Prompt 发给 LLM，让它基于检索结果写回答

所以 RAG 本质上是：**先查资料，再回答问题**。就像考试开卷——不准瞎写，必须引经据典。

### 4. 什么是向量（Vector/Embedding）？

这是最容易被问到的概念。想象一个坐标系：

- "猫"可能被映射到 [0.8, 0.5, 0.3, ...]
- "狗"可能被映射到 [0.7, 0.5, 0.4, ...]
- "汽车"可能被映射到 [-0.2, 0.1, 0.9, ...]

"猫"和"狗"的向量很接近（因为它们都是宠物），而"汽车"的向量离它们很远（不相关）。

向量就是一堆数字，用来表示一段文字"在语义空间的位置"。语义相似的文字，向量也相似。

你项目里的 `all-MiniLM-L6-v2` 模型就在做这件事——把文字变成向量。

### 5. 什么是向量数据库（ChromaDB）？

普通数据库存的是"一行行数据"。向量数据库存的是"向量"，支持"查找最相似"的搜索。

**类比**：
- 普通数据库：`SELECT * FROM books WHERE title = '人工智能'`（精确匹配）
- 向量数据库：`找到与"什么是深度学习"最相似的 5 个段落`（语义相似）

你项目里的 ChromaDB 就是向量数据库，你问一个问题，它找到意思最接近的段落。

### 6. 什么是 Agent（智能体）？

**一句话**：Agent = 能自主做决策的 AI。

你手机上的 Siri 不是 Agent——它只是你说啥它做啥。

真正的 Agent 会：
- **感知**：接收用户的自然语言输入
- **决策**：统一走检索优先策略，LLM 自主判断相关性（`retriever.py`）
- **执行**：检索知识库 + LLM 生成回答，不相关则 LLM 兜底
- **记忆**：记住之前的对话（`memory_manager.py`）

你的项目就是一个 Agent——所有问题先查知识库，查到就引用，查不到就 LLM 直接回答并标注来源。

### 7. 什么是 LangChain？

LangChain 是一个 Python 库，封装了构建 AI 应用常用的功能，让你不用从零写。

你用 LangChain 做了：
- `langchain_huggingface` → 调用嵌入模型
- `langchain_chroma` → 操作向量数据库
- `langchain_core.documents.Document` → 统一文档格式

### 8. 什么是检索优先 + LLM 兜底？

用户的所有问题，系统统一走一条流程：
1. 中英文双语检索知识库（`retriever.py`）
2. 检索结果注入 Prompt，LLM 自主判断相关性
3. 相关 → 基于知识库回答 + 引用来源
4. 不相关 → LLM 用自己的知识回答 + 标注"由大模型直接生成"

这样避免了以前"意图识别误判"导致的问题（比如"你是什么模型"被判为查知识库，返回"未找到"）。

### 9. 什么是对话记忆？

没有记忆的对话：
> 你："什么是 Python？"
> AI："一种编程语言。"
> 你："它有什么优点？"
> AI：**"你问的'它'是什么？"**

有记忆的对话：
> 你："什么是 Python？"
> AI："一种编程语言。"
> 你："它有什么优点？"
> AI：**"Python 语法简洁，库丰富。"** ← 知道"它"是 Python

你的 `memory_manager.py` 保存最近 10 轮对话，让 AI 能理解"它"、"那个"这类指代词。

## 第二部分：你的项目是怎么运转的

```
用户上传文档 (PDF/Word/TXT)
    ↓
document_loader.py   →  读取文档内容
    ↓
text_splitter.py     →  切成小段（每段500字）
    ↓
vector_store.py      →  每段→向量→存入 ChromaDB
    ↓
    知识库就绪 ✅
```

```
用户提出问题
    ↓
retriever.answer_with_fallback()  ←  统一入口
    ↓
双语检索（中文 + 英文关键词翻译）
    ↓
合并去重 → 注入 Prompt → LLM 自主判断
    ↓
┌─ 相关 ────────────────────────┐  ┌─ 不相关 ─────────────────┐
│ RAG 回答 + 引用来源            │  │ LLM 直接回答 + ⚠️标注     │
└───────────────────────────────┘  └──────────────────────────┘
    ↓
evaluation.py →  记录这条问答到日志
    ↓
返回答案给用户
```

## 第三部分：常见面试问题速答

### Q1：你的项目用了什么技术栈？
"Python + LangChain + ChromaDB + DeepSeek API。前端用 Streamlit 搭建。嵌入模型用的是 sentence-transformers 的 all-MiniLM-L6-v2，本地运行、不花钱。"

### Q2：为什么要用 RAG？LLM 自己不能回答吗？
"LLM 的知识有截止日期，而且不知道我的私有文档。RAG 把'检索'和'生成'解耦——先精确找到相关段落，再让 LLM 基于段落回答，相当于给 LLM 一本参考书让它照着回答，大大减少胡编乱造（幻觉）。"

### Q3：文本为什么要分块？不能整篇塞进去吗？
"每一块是一个独立的检索单元。如果整篇文章塞成一个向量，语义太模糊了。切成小块后，每个块代表一个具体的知识点，检索更精准。500 字和 50 字重叠是我设置的参数——重叠保证不会恰好把一句话切断。"

### Q4：意图识别为什么要用 LLM？不能用关键词匹配吗？
"关键词太死板了。'给我讲讲 RAG' 和 'RAG 是啥' 关键词不一样，但意思一样。LLM 理解的是语义而不是关键词。而且我的项目里意图识别用的是 low temperature（0），保证结果稳定。"

### Q5：对话记忆怎么实现？
"我用一个 ConversationMemory 类，内部维护一个消息列表，最多保留 10 轮（20 条消息）。每次新消息进来，自动把最旧的移出去，防止记忆过长撑爆上下文窗口。"

### Q6：向量数据库为什么选 ChromaDB？
"因为是轻量级的，一个 pip install 就能用，不需要装 Docker 或服务端。数据存在本地文件夹里，适合演示和小项目。生产环境可以考虑 Milvus 或 Pinecone。"

### Q7：评测日志有什么用？
"记录每条问答的意图类型、检索片段数、响应延迟。通过分析日志可以发现：哪些类型的问题检索效果差？意图识别准确率多少？响应时间是否稳定？这对应 JD 里提到的'自动化评测与回测机制'。"

### Q8：如果要改进，你会做什么？
"几个方向：① 加上多路检索（关键词 + 向量混合检索）提升召回率；② 实现多轮 Query 改写，比如用户问'那它有什么缺点'，系统自动补全为'XXX 有什么缺点'；③ 加入用户反馈机制（点赞/点踩）做强化学习；④ 支持更多文档格式（网页、Markdown 等）。"
